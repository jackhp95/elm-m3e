# Spec — B1.5c: Converter child-routing fix + cheap attr buckets

**Date:** 2026-07-07
**Branch:** `docs/b1.5c-child-routing`
**Depends on:** B1.5b (merged `104a2ff` — `M3e.StepperNext`/`M3e.FabMenuItem` bindings exist).

## Problem

Composite components (Tabs, Stepper, Tree, Select, RichTooltip, Step) ship **HTML/bottom-only**
(partial API toggle, strict `M3e` tab suppressed) because the HTML→Elm converter mis-routes their
children. Investigation found **three independent converter-correctness defects**, all of which must
be fixed for full 4-tab parity:

1. **The oracle can't see the two new tags.** `docs/scripts/examples-gen/lib/oracle.mjs` builds its
   tag→module map from the RAW `@m3e/web@2.5.13` CEM (`docs/node_modules/@m3e/web/dist/custom-elements.json`),
   keyed by `d.tagName` (`oracle.mjs:69`). That CEM has the same analyzer defect B1.5b fixed in the
   library generator: `M3eStepperNextElement.tagName="m3e-stepper-previous"` and
   `M3eFabMenuItemElement.tagName="m3e-menu-item"`. The oracle never goes through elm-cem's reconcile,
   so `oracle["m3e-stepper-next"]` / `oracle["m3e-fab-menu-item"]` are never created → those tags
   degrade with `unknown m3e tag …` at ALL surfaces (`examples.skipped.txt:76-80`). This is why
   Stepper/FabMenu examples degrade despite the Elm bindings now existing.

2. **Default-slot children are routed only by the `slot=` attribute, ignoring tag.** In real m3e
   markup, panels/steps sit in the DEFAULT slot with NO `slot=` attr, distinguished only by tag
   (`<m3e-tab-panel>` vs `<m3e-tab>`). `elementToElm` (`to-elm.mjs:353-394`) lumps every no-`slot=`
   child into one `M3e.<mod>.children [ … ]`, producing a heterogeneous `List Content` that fails to
   typecheck → *"The 1st argument to `children` is not what I expect."* Fix needs per-container
   child-tag→slot routing.

3. **The default-slot config key is read under the wrong name.** `oracle.mjs:166` looks up
   `slotConfig["default"]`, but `config/slots.json` uses the key `"unnamed"`. So `required`/`multi`/
   `kinds` for the default slot of IconButton/Heading/Chip are silently dropped, the `content` fold
   never fires, and the converter emits a non-existent `M3e.IconButton.child` (this also breaks
   SplitButton, which nests an icon-button). **This bug is also the root cause of the 3 "pre-existing"
   failures in `to-elm.test.mjs` (tests 4, 6, 18)** — the handoff listed them as unrelated; they are
   in fact this exact defect, and fixing it turns them green.

Separately, a few misc unmapped attrs (`m3e-toc-ignore`, `hidden`, `autofocus`) leak into skips.

## Fix

Four changes across `oracle.mjs` and `to-elm.mjs` (no library/`Generate.elm` changes):

### Fix A — tagName reconciliation in the oracle (`oracle.mjs`)
Mirror B1.5b's authoritative correction when reading the CEM: for every module export of kind
`custom-element-definition`, overwrite the referenced class declaration's `tagName` with the export's
`name` (the `customElements.define` registration truth) BEFORE the oracle keys entries by tagName.
No-op wherever the analyzer already agrees. This makes `m3e-stepper-next`/`m3e-fab-menu-item`
resolvable (their modules expose `view` + child/children/setters — verified present from B1.5b).

### Fix B — default-slot config-key bug (`oracle.mjs:166-167`)
Look up the default-slot config under `"unnamed"` (the real key) as well as `"default"`, and ensure
the CEM-slot loop does not pre-seed the empty-name slot with an empty config that short-circuits the
config-merge loop (`oracle.mjs:216-219`). After the fix, IconButton/Heading/Chip default slots carry
`required/multi/kinds` from `config/slots.json`, the `content` fold fires, and the 3 `to-elm.test.mjs`
failures pass.

### Fix C — child-tag routing by produced kind (`oracle.mjs` + `to-elm.mjs`)
Route a no-`slot=` default child to the correct NAMED slot when the child's produced kind matches a
container slot's kind. The oracle already knows each slot's accepted `kind` (`entry.slots[].kind`) and
each child tag's module (hence its produced kind). Derive a per-container map `childKind → slotHelper`
from the container's named slots, exposed as a new oracle field (e.g. `entry.childSlotByKind`). In
`to-elm.mjs`, at the default-slot fallback seam (`~line 383`): if the child has no `slot=` attr but its
produced kind maps to a named slot in the container, emit `M3e.<mod>.<helper> (<child>)` (a single
Content into `slottedExprs`) instead of the catch-all `children`. Otherwise keep current behavior
(default `child`/`children`).

**This derivation must reproduce the authoritative routing table** (memory `docs-composite-routing-table`):
- Tabs: `m3e-tab`→`children`/`child`; `m3e-tab-panel`→`panel`.
- Stepper: `m3e-step`→`step`; `m3e-step-panel`→`panel` (no `children`).
- Tree/TreeItem, NavMenuItem(+Group), Step/StepPanel, RichTooltip, Select/Option — route named-kind
  children to their slot helpers; leave the default union to `children`.
- Union-row components (Menu, MenuItemGroup, NavMenu, FabMenu for `m3e-menu-item`) already work via a
  union `children` row — the derivation must NOT pull those into named slots (their kinds are the
  container's default-union kinds, with no competing named slot). Verify these stay unchanged.

### Fix D — cheap attr buckets (`to-elm.mjs:61-65`, `isDroppableAttr`)
- Add `m3e-toc-ignore` to `isDroppableAttr` (docs-site marker, no Elm setter → drop).
- `hidden` / `autofocus`: if the component already exposes a typed setter for the attr
  (`entry.attributes.find(a => a.htmlName === name)`), the existing attr path maps it — leave that.
  Where no setter exists, drop (there is NO universal non-aria setter module — do not invent one).

## Out of scope (leave as graceful-degrade — genuine API gaps)
- `m3e-optgroup` inside `m3e-select` (no `Select.optgroup` helper; `Select.children` takes only
  `{option}`).
- Menu submenus (`submenu` is an `Attr Bool` + separate `<m3e-menu>` via MenuTrigger; no nested-child
  path).
These already ship at HTML/bottom via graceful degradation — do NOT force them (deferred library work).

## Acceptance criteria

   green; no regressions). `extract-matraic-examples.test.mjs` still 3/3.

   the composite examples whose degradation was a *converter child-routing* miss: **Tabs, Tree,
   Select (basic)** — spot-check each has a real Elm `M3e.<Composite>.view …` string.
   **Reclassified as deferred library-typing gaps (2026-07-07, post-review):** **Stepper** and
   **SplitButton** remain degraded not from routing but from genuine library typing gaps beyond this

   split-button's slotted `leading-button`/`trailing-button` + nested menu-trigger typing has no path.
   Both were null at baseline (no regression). They join the "Out of scope — genuine API gaps" list;
   fixing them is future library work, not B1.5c.
3. `m3e-stepper-next` and `m3e-fab-menu-item` no longer appear as `unknown m3e tag` skips in
   `config/examples.skipped.txt`; the examples using them regain a non-null `top`.
4. **Total example count holds or rises** vs. the pre-change baseline (capture the baseline count
   first; degradation invariant from B1.5a stays — no example is DROPPED, only upgraded).
5. Union-row composites (Menu/NavMenu/FabMenu) are **unchanged** in output (diff their emitted `top`).
6. `cd docs && npm run build:ci` → **exit 0** (`check-nav: OK`, `Success - Adapter script complete`).
7. `elm-format --validate app src` clean (from `docs/`) if any `.elm` hand-edited (none expected —

8. **Playwright, mobile + desktop viewports:** on a composite component page (e.g. `/components/tabs`

## Risks
- **R1 — kind-derivation misroutes.** A container whose `children` union kind ALSO has a dedicated
  named slot would be ambiguous. Mitigation: prefer the named slot only when the kind is NOT part of
  the container's default `children` union; where a kind is in the union (Menu family), keep it in
  `children`. Verify the derived map equals the authoritative table for every listed container before
  trusting it; if any container is ambiguous, fall back to an explicit per-tag entry for that one.
- **R2 — Fix B changes MORE than the 3 tests.** Restoring `required/kinds` on default slots could make
  other examples newly fold `content` or newly skip (missing required content). Mitigation: diff the
  full regen; every newly-skipped example must be a genuine "missing required slot" case, not a
  regression — quantify and justify.
- **R3 — Fix A resolves the 2 tags but they still don't compile** for want of a `config/slots.json`
  entry. Mitigation: if regen shows StepperNext/FabMenuItem examples failing `top` compile for a
  missing slot/required config, add the minimal `config/slots.json` entry mirroring a sibling; keep it
  minimal.
- **R4 — Count drops.** If any change causes a net loss of examples/surfaces, STOP and investigate;
  the B1.5a invariant (never drop, only degrade) is non-negotiable.
