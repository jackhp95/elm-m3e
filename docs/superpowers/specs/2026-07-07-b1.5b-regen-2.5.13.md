# Spec — B1.5b: Regenerate the Elm library from @m3e/web 2.5.13

**Date:** 2026-07-07
**Branch:** `docs/b1.5b-regen-2.5.13`
**Depends on:** nothing. **Unblocks:** B1.5c (needs `M3e.stepperNext` / `M3e.fabMenuItem` bindings).

## Problem

The generated Elm library `packages/m3e` (v1.0.0) is missing `M3e/StepperNext.elm` and
`M3e/FabMenuItem.elm`, even though `m3e-stepper-next` and `m3e-fab-menu-item` are real custom
elements in installed `@m3e/web@2.5.13`. Because of this, the docs example pipeline cannot bind
those two tags and their examples ship HTML/bottom-only.

## Root cause (verified 2026-07-07)

The bug is **not** version skew. It is a **CEM analyzer defect** in the shipped 2.5.13
`dist/custom-elements.json`: two class declarations carry the **wrong, colliding `tagName`**:

| Class (module)                                   | `tagName` in CEM (WRONG) | Correct tag (from `@customElement` decorator + `custom-element-definition` export) |
|--------------------------------------------------|--------------------------|-----------------------------------------------------------------------------------|
| `M3eStepperNextElement` (`src/stepper/StepperNextElement.ts`) | `m3e-stepper-previous` | `m3e-stepper-next` |
| `M3eFabMenuItemElement` (`src/fab-menu/FabMenuItemElement.ts`) | `m3e-menu-item` | `m3e-fab-menu-item` |

- The matraic source decorators are **correct** (`@customElement("m3e-stepper-next")` etc.), so
  this is analyzer output corruption, not a source bug.
- The **authoritative tag→class mapping survives** in each module's `custom-element-definition`
  **export** (`{ kind:"custom-element-definition", name:"m3e-stepper-next", declaration:{ name:"M3eStepperNextElement" } }`).
- `Generate.elm` builds components from class declarations (`decl.customElement == Just True`,
  ~line 2927), reads `.tagName` from them, and **merges components sharing a tagName**
  (`mergeSameTagName`, ~line 3044). So today `StepperNext` is merged into `StepperPrevious` and
  `FabMenuItem` into `MenuItem` → the two modules are never emitted.

## Fix

Add a **general, authoritative `tagName` reconciliation** preprocessing pass in the elm-cem CLI
(`elm-cem/bin/elm-cem.js`), then regenerate `packages/m3e` from the installed 2.5.13 CEM.

The reconciliation makes the **`custom-element-definition` export the source of truth for a class's
tag** (it is the registration truth — `customElements.define(name, Class)`). For every module
export of kind `custom-element-definition`, set the referenced class declaration's `tagName` to the
export's `name`. This is a no-op wherever the analyzer already agrees, so it is safe and uniform
across all libraries elm-cem targets — not a per-component patch. (Aligns with the standing
preference: uniformity > per-component ergonomics.)

Stay on **installed 2.5.13** — do NOT drift to the matraic clone's 2.5.14. No version bump, no
`cem analyze` step, durable across `pnpm install`.

## Scope

**In:**
- New `reconcileTagNames` pass in `elm-cem/bin/elm-cem.js`, wired as the **innermost** arg
  transform (runs before `inlineTypeAliases`/`injectConfig`/`injectRuntime` so corrected tags flow
  through every downstream pass and into `Generate.elm`).
- Regenerate `packages/m3e/src` with the canonical command:
  ```
  node elm-cem/bin/elm-cem.js \
    --flags-from=docs/node_modules/@m3e/web/dist/custom-elements.json \
    --config-from=config/slots.json \
    --config-from=config/native-mdn.json \
    --config-from=config/examples.generated.json \
    --output=packages/m3e/src
  ```
- Regenerate the docs reference (`packages/m3e` → `docs/data/reference.json`) via `build:reference`.

**Out (do NOT touch):**
- No config changes to `config/slots.json` (unless the two new modules need slot declarations to
  compile — see risk R3; if so, that is a minimal, justified addition).
- No changes to `Generate.elm` logic (the fix is upstream, in the CLI preprocessing).
- The unrelated `elm-cem/cem-configs/` fixtures for other libraries.
- The docs renderer / converter (that's B1.5c).

## Acceptance criteria

1. `packages/m3e/src/M3e/StepperNext.elm` and `packages/m3e/src/M3e/FabMenuItem.elm` exist.
2. Barrel `packages/m3e/src/M3e.elm` exposes `stepperNext` and `fabMenuItem` (grep confirms).
3. `packages/m3e/elm.json` `exposed-modules` includes both new modules (auto-synced by the CLI).
4. `M3e.StepperPrevious` and `M3e.MenuItem` are **unchanged in tag** (`m3e-stepper-previous` /
   `m3e-menu-item`) — the merge collision is gone, both siblings survive independently.
5. `cd packages/m3e && elm make src/M3e.elm --output=/dev/null` compiles clean.
6. `cd docs && npm run build:reference` succeeds; `docs/data/reference.json` gains both components.
7. `cd docs && npm run build:ci` → **exit 0** (prints `check-nav: OK`, ends `Success - Adapter
   script complete`).
8. `elm-format --validate` clean on any hand-edited `.elm` (none expected; the JS CLI change needs
   no elm-format).
9. **Ripple reviewed:** the regen diff is inspected in full. Additive changes (2 new modules +
   barrel + elm.json + reference.json) are expected; any *non-additive* change to existing modules
   is explained or reverted.

## Risks

- **R1 — large/noisy regen diff.** If the committed `packages/m3e` was generated with a slightly
  different elm-cem or config, a fresh regen may reformat/reorder existing modules. Mitigation:
  diff `git status --stat` + spot-diff a few unchanged-intent modules; if the only semantic change
  is the two additions, accept; otherwise investigate before merge.
- **R2 — reconcile over-reach.** A `custom-element-definition` whose `declaration` re-exports from
  another module. Mitigation: resolve `declaration.module` when present, else match by name across
  modules; if a class is targeted by two definitions (shouldn't happen), warn and take the last.
- **R3 — new modules fail to compile** for want of a slot/config entry. Mitigation: if `elm make`
  fails only on the two new modules, add the minimal `config/slots.json` entry they need (mirroring
  their sibling `StepperPrevious`/`MenuItem`) and regen; keep the addition minimal and justified.
- **R4 — reconcile changes OTHER tags** unexpectedly (analyzer wrong elsewhere too). This is
  *correct* behavior but could enlarge the diff. Mitigation: log every tagName the pass rewrites;
  confirm each rewrite is a genuine correction against the source decorator / export.
