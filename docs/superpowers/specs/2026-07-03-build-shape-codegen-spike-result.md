## Evidence — dup-singular fails compilation

Test `DupTest.elm`:
```elm
B.button { content = fakeContent, action = fakeAction }
    |> B.variant fakeVariant
    |> B.variant fakeVariant
```
Compiles to:
```
-- TYPE MISMATCH -----------------------------------------
The argument is:
    B.Builder { variant : M3e.Build.Internal.Used } B.SlotCaps msg
But (|>) is piping it to a function that expects:
    B.Builder { variant : M3e.Build.Internal.Available } B.SlotCaps msg
```

The dup-singular guarantee holds structurally. The error surfaces the exact
field state (`Used` vs `Available`) that caused the rejection.

## Key operational learnings for workstream B

These are the moving parts the generator emitter (`generateBuildModule`) must
handle correctly:

1. **Phantom type parameters via `Elm.customTypeWith`.** The
   `customTypeWith name generics variants` form takes an explicit `generics`
   list. `Elm.customTypeWith "Builder" ["attrCaps", "slotCaps", "msg"] [ … ]`
   produces the full phantom-parameterized type even though only `msg`
   appears in the variant argument.
2. **`Type.namedWith` preserves aliases.** Passing
   `Type.namedWith ["M3e", "Build", "Button"] "AttrCaps" []` inside a
   `withType` annotation emits the alias reference in `docs.json`, not the
   inlined record — this is what keeps the seed signature small.
3. **`Elm.Arg.customType "Builder" identity |> Elm.Arg.item (Elm.Arg.var "f")`**
   correctly emits the `(Builder f)` destructuring pattern for setter arg
   destructuring.
4. **External-module constructors need `Elm.value` with `importFrom`, not
   `Elm.val`.** `Elm.val` produces an unqualified reference (broken if the
   generated module doesn't `exposing` that constructor).
5. **Every exposed decl needs `|> Elm.withDocumentation "…"`.** Missing docs
   ⇒ `NO DOCS` compile error. Per component the emitter will chain docs on:
   Builder, AttrCaps, SlotCaps, button seed, N optional-attr setters, M
   optional-slot setters, `build`. Plus Internal's `Available`/`Used`.
6. **Error messages on dup-application** identify the offending field via
   `Available` vs `Used`. Acceptable UX; documentation should explain the
   vocabulary once so users can parse errors quickly.

## What was NOT needed

- The `Elm.Arg.var` + `withType`-covers-all workaround from
  `Generate.elm:2002-2009` was **not needed** for the ⑤ seed. That scar is
  specific to a 3-argument function shape (required record + attrs + content
  all annotated); the ⑤ seed's 1-argument shape doesn't trip the bug.
- No collapse from two-axis `Builder attrCaps slotCaps msg` to single-axis
  was required.
- No `M3e.Build.Aliases.elm` hand-written import workaround was required —
  codegen emits aliases correctly.

## Spike source shape

The runnable `Spike.elm` (≈330 lines) is a bounded elm-codegen driver whose
key surface is the pattern for **any** ⑤ setter. Representative excerpt (the
`variant` consuming-row setter emitter — the exact pattern the production
generator will apply per optional-singular attr):

```elm
variantSetterDecl : Elm.Declaration
variantSetterDecl =
    let
        inputBuilder =
            builderType
                (Type.extensible "a" [ ( "variant", available ) ])
                (Type.var "s")
                (Type.var "msg")

        outputBuilder =
            builderType
                (Type.extensible "a" [ ( "variant", used ) ])
                (Type.var "s")
                (Type.var "msg")

        fullType =
            Type.function [ valType, inputBuilder ] outputBuilder
    in
    Elm.declaration "variant"
        (Elm.fn2
            (Elm.Arg.var "v")
            (Elm.Arg.customType "Builder" identity
                |> Elm.Arg.item (Elm.Arg.var "f")
            )
            (\v f ->
                Elm.apply (Elm.val "Builder")
                    [ Elm.updateRecord [ ( "variant", Elm.just v ) ] f ]
                    |> Elm.withType outputBuilder
            )
            |> Elm.withType fullType
        )
        |> Elm.expose
        |> Elm.withDocumentation "Set the variant. Consumes the `variant` attr capability."
```

`icon` (slot) is identical structurally with `s`/`a` swapped and the field
name substituted. Every optional-singular field emits from the same template.
